    **Error types (src/api/errors.ts):**

    Create custom error classes that classify FFBaD API failures. These map directly to the user's error presentation decisions:
    - `NetworkError` — "Pas de connexion internet" (no network / timeout)
    - `ServerError` — "Le serveur FFBaD est indisponible" (5xx from FFBaD)
    - `AuthError` — Invalid credentials or session expired
    - `SchemaValidationError` — API returned unexpected shape (wraps ZodError)
    - `RateLimitError` — Extends ServerError, 429 responses (hidden from user)

    Each error class should:
    - Extend a base `FFBaDError` class
    - Have a `code` string property for programmatic matching
    - Have a `userMessageKey` string property pointing to an i18n key (e.g., 'auth.networkError')
    - Have `isRetryable: boolean` — NetworkError and RateLimitError are retryable; AuthError is not

    **Zod schemas (src/api/schemas.ts):**

    Create Zod schemas for FFBaD API responses. The API returns JSON with a `Retour` field:

    Base response wrapper:
    ```typescript
    const FFBaDResponseSchema = <T extends z.ZodType>(dataSchema: T) =>
      z.object({
        Retour: dataSchema,
      });
    ```

    Specific schemas for Phase 1 endpoints:

    `AccountPoonaSchema` — for `ws_getaccountpoona`:
    ```typescript
    z.object({
      Retour: z.union([
        z.array(z.object({
          Licence: z.string(),
          Nom: z.string(),
          Prenom: z.string(),
          // Add fields as discovered during testing
        })),
        z.string(), // Error messages come as strings
      ]),
    })
    ```

    `LicenceInfoSchema` — for `ws_getlicenceinfobylicence`:
    ```typescript
    z.object({
      Retour: z.array(z.object({
        Licence: z.string(),
        Nom: z.string(),
        Prenom: z.string(),
        Club: z.string().optional(),
        IS_ACTIF: z.union([z.boolean(), z.number()]).optional(),
      })),
    })
    ```

    Make schemas permissive with `.passthrough()` on objects — we don't want to fail on unknown fields the API adds. Use `.optional()` generously since the API may not always return all fields.

    ```typescript
    export type LicenceInfo = z.infer<typeof LicenceInfoSchema>;
    ```

    Also export the credential type:

  <action>
    **API client (src/api/client.ts):**

    Create an axios instance configured for the FFBaD RPC-over-REST API:

    Base configuration:
    - `baseURL`: `'https://api.ffbad.org/rest/'` (HTTPS enforced — no HTTP fallback)
    - `timeout`: 15000ms
    - `headers`: `{ 'Content-Type': 'application/json' }`

    The client needs a way to receive credentials. Export a `setCredentials` function that stores credentials in module-level state (NOT in SecureStore — that's the auth layer's job). The request interceptor reads from this state.

    **Request interceptor:**
    - Read current credentials from module state
    - If credentials exist, add `AuthJson` parameter with `{ Login: licence, Password: password }` as JSON string
    - Add `QueryJson` parameter from the request's custom config

    **Response interceptor (error handler):**
    - No response at all (network error): throw `NetworkError`
    - Status 429: throw `RateLimitError`
    - Status >= 500: throw `ServerError`
    - Status 401/403: throw `AuthError`
    - Otherwise: pass through the raw error

    **Retry logic with exponential backoff:**

    Implement retry as an axios response interceptor or as a wrapper function:
    - Max retries: 3 (per user decision: "auto-retry silently 2-3 times")
    - Only retry on `isRetryable` errors (NetworkError, RateLimitError, ServerError)
    - Do NOT retry AuthError (wrong credentials should fail fast)
    - Backoff formula: `baseDelay * 2^attempt * (0.5 + random * 0.5)` where baseDelay = 500ms
    - This gives: ~500ms, ~1000ms, ~2000ms with jitter
    - Rate limiting is hidden from user (per decision) — just backs off silently

    Export:
    - `apiClient` — the configured axios instance
    - `setCredentials(credentials: FFBaDCredentials | null)` — update auth state
    - `callFFBaD<T>(params: FFBaDCallParams, schema: ZodType<T>): Promise<T>` — typed wrapper that builds QueryJson, calls API, validates with Zod

    The `callFFBaD` function:
    1. Builds QueryJson from function name + params
    2. Calls apiClient.get with QueryJson as param
    3. Parses response.data through provided Zod schema using `safeParse`
    4. If safeParse fails: throw `SchemaValidationError` with Zod issues
    5. If safeParse succeeds: return `result.data`

    **FFBaD function wrappers (src/api/ffbad.ts):**

    Create typed, named functions for each Phase 1 API endpoint:

    ```typescript
    export async function validateCredentials(
      licence: string,
      password: string
    ): Promise<AccountPoona> {
      // Temporarily set credentials for this call
      setCredentials({ licence, password });
      const result = await callFFBaD(
        { fonction: 'ws_getaccountpoona', params: [licence, password] },
        AccountPoonaSchema
      );
      return result;
    }

    export async function getLicenceInfo(
      licence: string
    ): Promise<LicenceInfo> {
      return callFFBaD(
        { fonction: 'ws_getlicenceinfobylicence', params: [licence, false] },
        LicenceInfoSchema
      );
    }
    ```

    Also export functions for future phases (stubs with proper typing):
    ```typescript
    export async function getResultsByLicence(licence: string) { /* Phase 4 */ }
    export async function getRankingEvolution(licence: string) { /* Phase 5 */ }
    ```

    IMPORTANT: The FFBaD API passes credentials as URL query parameters (HTTPS encrypts these in transit, but they may appear in server logs). This is the API's design constraint — document it in a code comment but don't try to work around it.
